# ADR 0023: Human-Readable Labels for Transformation Nodes

* **Status**: proposed
* **Deciders**: jakobmoellerdev
* **Date**: 2026-05-27

Technical Story: Transfer progress output shows opaque auto-generated camelCase IDs (e.g. `transformOcmSoftwareMyApp100Upload [AddComponentVersion]`) that are difficult to read and understand during a transfer operation.

## Context and Problem Statement

When `ocm transfer component-version` runs, each transformation node in the execution DAG is displayed in the progress output using its internal ID. These IDs are generated by `identityToTransformationID()` to be valid DAG vertex keys and are embedded in CEL expression strings (e.g. `${transformOcmSoftwareMyApp100Upload.output.resource}`). The output looks like:

```text
    ✓ transformOcmSoftwareMyApp100Upload [AddComponentVersion]
    ⠋ transformOcmSoftwareMyApp100GetMyapp [GetLocalResource]
    ✓ transformOcmSoftwareTest010AddIcons [AddLocalResource]
```

Users cannot easily understand which component and resource is being processed without mentally parsing the camelCase ID. The goal is to produce output like:

```text
    ✓ my-app@1.0.0 [Upload]
    ⠋ my-app@1.0.0 [Get myapp]
    ✓ test@0.1.0 [Add icons]
```

## Decision Drivers

* IDs used in CEL expressions and as DAG vertex keys must not change — any rename would break graph execution
* The change must be backwards-compatible with existing serialized `TransformationGraphDefinition` JSON (e.g. written to disk or exchanged between versions)
* Display logic should not be duplicated across the CLI and any future tooling consuming the graph
* The label format should be derivable purely from information already available at graph construction time (component name, version, resource name, operation type)

## Considered Options

* Option A: Change the IDs to be human-readable
* Option B: Add a separate `Label` field to `TransformationMeta`
* Option C: Derive the label only in the CLI layer, without touching the transformation metadata

## Decision Outcome

Chosen [Option B](#option-b): add a `Label` field to `TransformationMeta`.

Justification:

* IDs cannot change (Option A excluded) — they are embedded in CEL strings and act as DAG keys
* Putting the label in the metadata means any future consumer (other CLIs, GUIs, log formatters) can use it without re-deriving it from the ID
* The CLI formatter simply prefers `Label` when non-empty and falls back to the existing `ID [Type]` format, requiring minimal change at the display layer

### Option B

#### Description

Add an optional `Label string` field to `TransformationMeta` with `json:"label,omitempty"`. The field is set at graph construction time in `bindings/go/transfer/internal/` and consumed by the CLI progress formatter in `cli/cmd/transfer/component-version/progress.go`.

#### Label Format Specification

| Transformation | Format |
|---|---|
| Upload (AddComponentVersion) | `{shortName}@{version} [Upload]` |
| Get resource | `{shortName}@{version} [Get {resourceName}]` |
| Add resource | `{shortName}@{version} [Add {resourceName}]` |
| Convert resource (Helm) | `{shortName}@{version} [Convert {resourceName}]` |
| Cleanup | `Cleanup temp files` |

Where `shortName` is the last `/`-separated segment of the component name (e.g. `ocm.software/my-app` → `my-app`).

For components with multiple targets, append `→ target N` (0-indexed) to disambiguate:

```text
my-app@1.0.0 [Upload → target 0]
my-app@1.0.0 [Upload → target 1]
```

#### Contract

```go
// TransformationMeta contains metadata for a transformation.
type TransformationMeta struct {
    Type  runtime.Type `json:"type"`
    ID    string       `json:"id"`
    // Label is an optional human-readable display name used by progress
    // renderers. Purely cosmetic — ID remains the canonical identifier
    // referenced by CEL expressions and DAG vertex keys.
    // When empty, consumers fall back to ID-based formatting.
    Label string       `json:"label,omitempty"`
}
```

CLI formatter:

```go
func formatTransformationName(t *graphPkg.Transformation) string {
    if t.Label != "" {
        return t.Label
    }
    return fmt.Sprintf("%s [%s]", t.ID, t.Type.Name)
}
```

## Pros and Cons of the Options

### [Option A] Change IDs to Human-Readable Strings

Pros:

* Single source of truth — ID and display name are the same
* No extra field in the metadata struct

Cons:

* Breaks CEL expressions: all `${...}` references in generated graph specs would need to be regenerated
* Breaks DAG determinism if IDs are not globally unique (two resources named `myapp` in different components)
* Breaking change for any persisted or exchanged `TransformationGraphDefinition` JSON

### [Option B] Add `Label` Field to `TransformationMeta`

Pros:

* No change to IDs, CEL expressions, or DAG keys
* Label is available to all consumers of the metadata, not just the CLI
* Backwards compatible via `omitempty`
* Fallback to existing format when label is absent

Cons:

* Slight redundancy: two representations of the same node
* Requires codegen rerun after field addition

### [Option C] Derive Label Only in CLI Layer

Pros:

* No change to `bindings/go/` modules
* Simpler diff

Cons:

* The derivation logic (parsing camelCase IDs back to names) is fragile and lossy
* Any other consumer has to re-implement the same logic
* Tightly couples the CLI to internal ID generation conventions

## Discovery and Distribution

Implementation proceeds across four PRs to respect module boundaries in this multi-module repository:

1. **PR 1** (this ADR): no code changes
2. **PR 2**: Add `Label` field to `TransformationMeta` in `bindings/go/transform/spec/v1alpha1/meta/meta.go`; rerun codegen (`task generate`)
3. **PR 3**: Set `Label` during graph construction in `bindings/go/transfer/internal/` (after PR 2 is merged and the dep is bumped); add unit tests asserting label values
4. **PR 4**: Use `Label` in `formatTransformationName` in `cli/cmd/transfer/component-version/progress.go` (after PR 3)

## Conclusion

Adding a cosmetic `Label` field to `TransformationMeta` is the smallest backwards-compatible change that makes transfer progress output immediately readable to users while leaving internal DAG mechanics untouched.
